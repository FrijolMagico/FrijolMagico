## Próximo Paso de Ejecución: Normalización y Carga

### 1. Migración de Base de Datos (Schema Update)

Los estados actuales (`seleccionado`, `confirmado`, `cancelado`, `no_asistio`) son insuficientes. Se requiere modificar las tablas `evento_edicion_participante` y `participante_exposicion` para soportar la nueva lógica de negocio:

| Nivel         | Estado         | Descripción                                     |
| :------------ | :------------- | :---------------------------------------------- |
| **Global**    | `renuncia`     | Artista se retira voluntariamente del evento.   |
| **Global**    | `expulsado`    | Organización remueve al artista (veto).         |
| **Global**    | `confirmado`   | Artista participa activamente.                  |
| **Global**    | `seleccionado` | Artista seleccionado pero no confirmado aún.    |
| **Instancia** | `desistido`    | Artista no realiza la actividad específica.     |
| **Instancia** | `cancelado`    | Organización suspende la actividad (logística). |
| **Instancia** | `ausente`      | Artista no se presenta.                         |
| **Instancia** | `confirmado`   | Actividad confirmada.                           |

_Nota: Al ser SQLite, esto requiere recrear las tablas o modificar los constraints CHECK._

### 2. Preparación de Datos (Script SQL)

1.  **Nuevos Artistas**: Insertar en `artista` (Pinkariine, Doppimine, LecA, Fluchinick).
2.  **Participaciones**: Insertar en `evento_edicion_participante` (63 confirmados + 4 renuncias).
3.  **Exposiciones**: Insertar en `participante_exposicion` (63 confirmados).

### 3. Ejecución
